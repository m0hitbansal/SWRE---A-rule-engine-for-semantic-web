Rule Engine

The rule engine creates custom rules which are to be executed using one of the chaining paradigms and update the knowledge base. By default, the system operates on forward chaining.

Operating the Rule Engine

Select the existing rules you want to execute on the selected ontology by clicking on the check box against that rule and hit Submit. Submitting will trigger the
Forward Chaining operating on default. If this is the first time you are operating on a fresh ontology, the forward chaining shall take a little more time than usual to update the knowledge base as per the implicit rules specified by the onotology i.e. updating the knowledge base on the basis of the constraints provided implicitely by the ontology.
Note: Forward chaining in a computationally expensive. It runs same rule in multiple passes until no new fact is added to the knowledge base. For simple boolean single level queries, try backward chaining instead of foward chaining to obtain quicker results.
Backward Chaining The backward chaining paradigm can be located on the Query tab. For quick results on the literal based queries, prefer backward chaining.

Create A New Rule

Each rule in the rule engine comprises two separate parts i.e. the antecedent and the consequent.
Antecedent: An antecedent rule can be a combination of several triples connected by an AND/OR connector.
Literals can also be inserted in the subject and object part of the triple which forms the building block of the antecedent. For adding a literal, select the "Other" field in the subject/object, a text box shall appear, fill in the required literal value.
Predicate section of the antecedent consists all the object properties, data properties and sub-properties that exist in the current ontology The current subjects and objects selected in the antecedent part are the only ones which could form the subjects and objects in the consequent Press the Done button to freeze the composite antecedent rule.
Consequent: A consequent rule is a triple. The objects/subjects selected in the antecedent are the only objects/subjects available in the consequent part. The predicate is a text field. The predicate will take existing object properties, data properties and sub-properties as an input. Please keep a check on the case-sensitivity. New predicates can also be added in this section which will update the knowledge base.

Note: The literals added to the antecedent part shall also appear in the consequent. Please refrain yourself to use a literal which is not present in the ontology i.e. do not add a literal using the rule engine as it would not contain the data properties used to define as in the ontology. We would advice you to update the ontology if you want to add new literals and work on them
Key Note: If two different versions of the same class are to be used, example ?Student isBatchmateOf ?Student, select the "Other" option from the drop down menu and input ?Student1 to generate the desired results.
